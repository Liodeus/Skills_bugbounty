# WAF Fingerprints — long-tail product lookup

Reference table for `/waf-bypass` **Phase 2** (and recon's **B.1** quick set). The compact
inline table in `SKILL.md` covers the ~10 products you'll hit in 2026; this file is the long tail
for the exotic CDN / edge / regional / CMS-plugin case.

**How to read a row:** *Ease* = how hard the product is to fingerprint (Easy / Moderate / Difficult).
*Fingerprint* lists the signals — status code, response **header**, **cookie**, or block-page **body**
snippet — case-insensitive where noted. Most rows come from the Awesome-WAF corpus; status codes and
header names are the highest-signal indicators (check those first).

> **Passive before active.** Read these from the responses recon already fetched (`curl` in step B) —
> headers/cookies/body — before you ever send a noisy probe. A confirmed header fingerprint needs no
> provocation.

## Cross-cutting tells (product-agnostic)

- **Unusual status code** on a malicious request (`406`, `418`, `451`, `493`, `999`, `503`) where the
  origin would return `200`/`404` → a WAF is intercepting. Map the exact code back to the table above.
- **Jumbled/misspelled headers** (`X-Cnection`, `nnCoection`, `X-WA-Info`) → a proxy is rewriting —
  usually BIG-IP / NetScaler.
- **Cookie set on first request with no login** (`incap_ses`, `__cf_bm`, `_abck`, `AWSALB`,
  `__jsluid`) → CDN/edge bot-management layer fronting the WAF.
- **Block page is a generic 403/404 with no server banner** → likely ModSecurity-CRS or a CDN in
  custom-block mode; fall back to the noisy-probe + timing tells in `SKILL.md` Phase 1.
